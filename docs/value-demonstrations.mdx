# Why Composable Skills?

This document shows **concrete value** through real examples. No theory—just practical benefits.

## Problem 1: Type Mismatches at Runtime

### Before: Silent Failures

```python
# Traditional approach: hope the data matches
def review_pr(pr_number):
    pr_data = fetch_pr(pr_number)           # Returns dict
    diff = get_diff(pr_data["id"])          # Expects string ID
    analysis = analyze_code(diff)            # Expects specific format
    return format_review(analysis)           # Hopes format is right
```

**What goes wrong:**
- `pr_data["id"]` might be an integer, not string
- `diff` format varies by Git provider
- `analysis` structure assumptions break silently
- Errors surface at runtime, in production, with real users

### After: Validated at Definition Time

```yaml
# pr-read outputs:
outputs:
  - name: pr_id
    type: string      # Contract: always string
    required: true

# pr-diff-read inputs:
inputs:
  - name: pr_id
    type: string      # Must match
    required: true
```

**Validation catches:**
```bash
$ pytest tests/test_skill_validation.py -v
FAILED: pr-diff-read expects string pr_id, but pr-read outputs integer
```

**Result:** Errors caught before any code runs.

## Problem 2: Monolithic, Untestable Tools

### Before: All-or-Nothing Testing

```python
def complete_pr_workflow(pr_number, reviewer_id):
    """Does everything, tests nothing individually."""
    pr = github.get_pr(pr_number)
    diff = github.get_diff(pr.id)

    # 50 lines of review logic
    issues = analyze_for_issues(diff)
    suggestions = generate_suggestions(issues)

    # 30 lines of comment posting
    for suggestion in suggestions:
        github.post_comment(pr.id, suggestion)

    # 20 lines of notification
    slack.notify_reviewer(reviewer_id, pr.id)
    email.send_summary(pr.author, issues)

    return {"status": "done", "issues": len(issues)}
```

**Testing requires:** Mock GitHub API, mock Slack, mock email, hope integration works.

### After: Testable at Every Level

```yaml
# Level 1: Test individually
_atomic/pr-read/            # Test: does it parse PR correctly?
_atomic/pr-diff-read/       # Test: does it extract diff correctly?
_atomic/code-analyze/       # Test: does analysis find issues?

# Level 2: Test composition
_composite/pr-review/       # Test: do L1s combine correctly?

# Level 3: Test workflow
_workflows/pr-review-complete/  # Test: does state machine work?
```

**Test pyramid:**
```
         /\
        /  \     3 workflow tests
       /----\
      /      \   12 composite tests
     /--------\
    /          \ 42 atomic tests
   --------------
```

**Result:** 684 tests across 137 skills. Each level isolated.

## Problem 3: No Safety Guarantees

### Before: Hope Nothing Dangerous Happens

```python
def process_request(user_input):
    # What could go wrong?
    result = llm.execute(user_input)  # Could do anything
    return result
```

**Risks:**
- LLM might call destructive APIs
- No visibility into what operations occurred
- User has no chance to confirm dangerous actions

### After: Graded Safety Levels

```yaml
# READ skills: Always safe
pr-read:
  operation: READ
  # Never modifies anything

# WRITE skills: Require confirmation
pr-comment-write:
  operation: WRITE
  confirmation_required: true
  # User sees: "Post comment to PR #42? [y/n]"

# WORKFLOW: Confirms at write boundaries
pr-review-complete:
  operation: WORKFLOW
  # READ phases run freely
  # WRITE phases prompt for confirmation
```

**User experience:**
```
Agent: I'll review PR #42 now.
[Reads PR, analyzes diff, generates review - no prompts]

Agent: Ready to post 3 review comments. Proceed? [y/n]
User: y
[Posts comments]
```

**Result:** Users control when writes happen.

## Problem 4: Debugging Black Boxes

### Before: "Something Failed Somewhere"

```
Error: NoneType has no attribute 'title'
Traceback:
  File "workflow.py", line 247, in process
    ...
```

**Where did it break?** Which of the 15 operations failed? What was the input?

### After: Skill-Level Observability

```
Workflow: pr-review-complete
├─ pr-read (pr_number: 42) ✓
│  └─ Output: {title: "Fix auth bug", files: 3}
├─ pr-diff-read (pr_id: "42") ✓
│  └─ Output: {additions: 45, deletions: 12}
├─ code-analyze (diff: {...}) ✗
│  └─ Error: Timeout after 30s
│  └─ Input that failed: {diff: "...2000 lines..."}
```

**Result:** Exact skill, exact input, exact failure point.

## Problem 5: Duplicate Logic Everywhere

### Before: Copy-Paste Across Projects

```
project-a/tools/github.py    # PR reading logic
project-b/utils/github.py    # Same logic, slightly different
project-c/lib/gh_client.py   # Same again, with bugs
```

**Maintenance nightmare:** Fix a bug in one, forget the others.

### After: Compose, Don't Duplicate

```yaml
# One atomic, used everywhere
_atomic/pr-read/SKILL.md

# Portfolio manager uses it
portfolio-manager/_composite/code-risk-analysis/
  composes:
    - pr-read    # Same skill
    - risk-evaluate

# Agent orchestrator uses it
agent-orchestrator/_workflows/pr-review-complete/
  composes:
    - pr-read    # Same skill
    - pr-review
```

**Result:** One source of truth. Updates propagate.

```
Traditional approach:

  - Integration failures in production
  - Debugging requires full-stack traces

  - 35 errors caught at validation

  - Debugging at skill boundary
```

### Development Velocity

```
Before (monolithic tools):
  - New workflow: 2-3 days (build from scratch)
  - Testing: 4-6 hours (mock everything)
  - Bug fixes: 1-2 hours (find the right place)

After (composable skills):
  - New workflow: 2-4 hours (compose existing)
  - Testing: 30 minutes (run skill tests)
  - Bug fixes: 15 minutes (isolated to skill)
```

## Real-World Scenario

### Trip Planning: Before vs After

**Before:**
```python
def plan_trip(destination, dates, preferences):
    # 500 lines doing everything:
    # - Search flights
    # - Search hotels
    # - Check weather
    # - Find restaurants
    # - Book everything
    # - Send confirmations
    pass
```

**After (Trip Optimizer showcase):**
```yaml
_workflows/trip-optimizer/
  composes:
    - destination-research   # L2: Combines search + weather + events
    - transport-optimize     # L2: Flights + trains + rentals
    - accommodation-optimize # L2: Hotels + Airbnb + pricing
    - itinerary-create      # L2: Schedule + restaurants + activities

# Each L2 composes L1 atomics:
destination-research/
  composes:
    - web-search         # L1
    - weather-forecast   # L1
    - event-search       # L1
```

**Result:** 41 skills, 85 tests, infinitely recomposable.

## Summary

| Problem | Traditional | Composable Skills |
|---------|-------------|-------------------|
| Type safety | Runtime errors | Validation-time |
| Testing | All-or-nothing | Per-skill isolation |
| Safety | Hope for the best | Graded confirmations |
| Debugging | Stack trace archaeology | Skill-level traces |
| Reuse | Copy-paste | Compose |

The framework exists to make these benefits automatic. You focus on what skills do. The system ensures they work together correctly.
