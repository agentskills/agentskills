スキルは有効化されてはじめて役に立ちます。`SKILL.md` frontmatter の `description` フィールドは、エージェントが特定タスクでスキルを読み込むか判断する主な手掛かりです。説明が曖昧すぎると必要時に発火せず、広すぎると不要時に発火します。

このガイドでは、発火精度を上げるためにスキル説明を体系的に検証・改善する方法を説明します。

## スキル発火の仕組み

エージェントは [段階的開示](/specification#progressive-disclosure) でコンテキストを管理します。起動時は利用可能な各スキルの `name` と `description` だけを読み込み、関連性を判断できる最小情報だけ取得します。ユーザータスクが説明に一致したときにのみ、`SKILL.md` 全体を読み込んで手順に従います。

つまり、発火判定の多くを説明文が担います。説明文が「いつ有効か」を伝えられないと、エージェントはそのスキルを選べません。

重要な補足として、エージェントは通常「単体能力を超える知識・能力」が必要なときにスキルを参照します。たとえば「このPDFを読んで」のような単純依頼は、説明が一致していても基本ツールで処理できるためPDFスキルが発火しないことがあります。未知のAPI、業務特化ワークフロー、特殊フォーマットなど、専門性が必要なタスクで説明文の質が効いてきます。

## 効果的な説明文の書き方

テスト前に、良い説明文の原則を押さえておくと効果的です。

- **命令形で書く。** 「This skill does...」ではなく「Use this skill when...」のように、実行判断の指示として書きます。
- **実装よりユーザー意図に寄せる。** スキル内部の仕組みではなく、ユーザーが達成したいことを記述します。
- **やや積極的に書く。** ユーザーがドメイン名を直接書かないケースも含め、適用コンテキストを明示します（例: 「'CSV' や 'analysis' と明示されなくても」）。
- **簡潔に保つ。** 数文〜短い段落が目安です。範囲を十分にカバーしつつ、複数スキルでのコンテキスト肥大化を避けます。[仕様](/specification#description-field) では 1024 文字上限があります。

## 発火評価クエリの設計

発火を検証するには、評価クエリ集合が必要です。各クエリに「このスキルが発火すべきか」をラベル付けした、実際のユーザープロンプトに近いデータを用意します。

```json eval_queries.json
[
  {
    "query": "I've got a spreadsheet in ~/data/q4_results.xlsx with revenue in col C and expenses in col D — can you add a profit margin column and highlight anything under 10%?",
    "should_trigger": true
  },
  {
    "query": "whats the quickest way to convert this json file to yaml",
    "should_trigger": false
  }
]
```

目安は20件前後です。発火すべきクエリ 8〜10 件、発火すべきでないクエリ 8〜10 件を用意します。

### 発火すべきクエリ

説明文がスキル範囲を捉えているかを確認します。次の軸でバリエーションを持たせます。

- **言い回し**: 形式的、口語的、typo・略語ありを混在させる
- **明示性**: ドメイン名を直接書くもの（"analyze this CSV"）と、書かないもの（"このデータからグラフが必要"）を混ぜる
- **詳細度**: 短文プロンプトと文脈豊富な長文を混在させる
- **複雑度**: 単発タスクと複数ステップタスクを混在させ、大きな依頼の中で関連性を見抜けるか確認する

特に有用なのは、「スキルが有効だが、クエリだけでは結び付きが自明でない」ケースです。こうしたケースでは説明文の書き方が結果を左右します。クエリがスキル内容そのものを直接求める場合、通常はどんな説明文でも発火しやすくなります。

### 発火すべきでないクエリ

価値が高い負例は **near-miss** です。キーワードは似ているが実際には別作業を求めるクエリで、説明文が「広いだけ」でなく「正確」かを検証できます。

CSV分析スキルなら、弱い負例は次のようなものです。

- `"Write a fibonacci function"` — 明らかに無関係で検証価値が低い
- `"What's the weather today?"` — キーワード重なりがなく簡単すぎる

強い負例:

- `"I need to update the formulas in my Excel budget spreadsheet"` — spreadsheet/data は共通だが必要なのは Excel 編集であり CSV 分析ではない
- `"can you write a python script that reads a csv and uploads each row to our postgres database"` — CSV を含むが、タスクは分析ではなく DB ETL

### 現実的なクエリにするコツ

実際のユーザープロンプトには、汎用テスト文にない文脈があります。次を含めてください。

- ファイルパス（`~/Downloads/report_final_v2.xlsx`）
- 個人的文脈（`"my manager asked me to..."`）
- 具体情報（列名、企業名、データ値）
- 口語、略語、軽微な typo

## 説明文が発火するかの検証

基本方針は、スキルを導入した状態で各クエリを実行し、エージェントがそのスキルを呼び出したかを観察することです。スキルがクライアントで登録・探索可能になっていることを確認してください（設定方法はクライアントにより異なり、スキルディレクトリ、設定ファイル、CLI フラグなどがあります）。

多くのクライアントは、実行ログ、ツール呼び出し履歴、verbose 出力などの観測手段を提供しています。詳細は各クライアントのドキュメントを参照してください。エージェントが `SKILL.md` を読み込んでいれば発火、参照せず進んだ場合は未発火です。

A query の「合格」条件:

- `should_trigger` is `true` and the skill was invoked, or
- `should_trigger` is `false` and the skill was not invoked.

### 複数回実行

モデル挙動は非決定的です。同じクエリでも発火したりしなかったりします。各クエリを複数回（まずは3回が目安）実行し、スキルが呼び出された割合を **trigger rate** として計算します。

発火すべきクエリは trigger rate が閾値以上（デフォルト 0.5 推奨）で合格、非発火クエリは閾値未満で合格です。

20件を3回ずつ回すと60実行になるため、スクリプト化を推奨します。以下は雛形です。`check_triggered` 内の `claude` 呼び出しと判定ロジックは利用クライアントに合わせて置き換えてください。

```bash
#!/bin/bash
QUERIES_FILE="${1:?Usage: $0 <queries.json>}"
SKILL_NAME="my-skill"
RUNS=3

# This example uses Claude Code's JSON output to check for Skill tool calls.
# Replace this function with detection logic for your agent client.
# Should return 0 (success) if the skill was invoked, 1 otherwise.
check_triggered() {
  local query="$1"
  claude -p "$query" --output-format json 2>/dev/null \
    | jq -e --arg skill "$SKILL_NAME" \
      'any(.messages[].content[]; .type == "tool_use" and .name == "Skill" and .input.skill == $skill)' \
      > /dev/null 2>&1
}

count=$(jq length "$QUERIES_FILE")
for i in $(seq 0 $((count - 1))); do
  query=$(jq -r ".[$i].query" "$QUERIES_FILE")
  should_trigger=$(jq -r ".[$i].should_trigger" "$QUERIES_FILE")
  triggers=0

  for run in $(seq 1 $RUNS); do
    check_triggered "$query" && triggers=$((triggers + 1))
  done

  jq -n \
    --arg query "$query" \
    --argjson should_trigger "$should_trigger" \
    --argjson triggers "$triggers" \
    --argjson runs "$RUNS" \
    '{query: $query, should_trigger: $should_trigger, triggers: $triggers, runs: $runs, trigger_rate: ($triggers / $runs)}'
done | jq -s '.'
```

<Tip>
  クライアントが対応していれば、結果が確定した時点で早期終了できます（スキル参照の有無が判定できた時点）。評価全体の時間とコストを大きく削減できます。
</Tip>

## train/validation 分割で過学習を防ぐ

全クエリに対して説明文を最適化すると、過学習のリスクがあります。特定の言い回しには効くが、未知の表現には効かない説明文になる可能性があります。

対策はクエリセットを分割することです。

- **Train set（約60%）**: 失敗分析と改善方針に使う
- **Validation set（約40%）**: 改善が汎化しているか最終確認する

両セットで発火/非発火クエリの比率を揃えてください。片方に正例を偏らせないようにします。比較条件を揃えるため、分割はランダムに作って反復中は固定します。

[上記](#running-multiple-times) のようなスクリプトを使うなら、`train_queries.json` と `validation_queries.json` の2ファイルに分けて別々に実行できます。

## 最適化ループ

1. **評価**: 現在の説明文を _train/validation 両方_ で評価する。train は改善方針、validation は汎化確認に使う。
2. **失敗抽出**: _train set_ で、発火しなかった正例・誤発火した負例を特定する。

- 改善方針には train の結果だけを使い、validation 結果は混ぜない。

3. **説明文改訂**: 汎化を意識して修正する。

- 正例が発火しないなら説明が狭すぎる可能性。適用範囲や有効文脈を補う。
- 負例で誤発火するなら説明が広すぎる可能性。対象外を明示し、隣接スキルとの境界を明確化する。
- 失敗クエリの単語をそのまま足すのは避ける（過学習）。代わりに背後の概念カテゴリを抽象化して反映する。
- 何度か詰まったら微調整ではなく、文構造や観点を変える。
- 説明文は最適化中に長くなりやすいため、1024文字上限を維持する。

4. **反復**: 1〜3 を、train 全合格または改善頭打ちまで繰り返す。
5. **最良反復の選択**: _validation set_ の合格率で選ぶ。最後の版が最良とは限らず、過学習前の版が最良なこともある。

通常は5反復程度で十分です。改善しない場合は、説明文ではなくクエリ設計（難易度、ラベル品質）に課題がある可能性があります。

<Tip>
  [`skill-creator`](https://github.com/anthropics/skills/tree/main/skills/skill-creator)
  スキルを使うと、このループを end-to-end で自動化できます。評価セット分割、並列
  trigger rate 評価、Claude による説明改善提案、実行中に見られる HTML
  レポート生成まで対応します。
</Tip>

## 結果の適用

最良説明文を選んだら、次を実施します。

1. `SKILL.md` frontmatter の `description` を更新する。
2. [1024文字上限](/specification#description-field) を満たすことを確認する。
3. 期待どおり発火することを確認する。まずは手動で数件試し、さらに厳密に行う場合は新規クエリ 5〜10 件（正例/負例混在）で評価スクリプトを実行する。最適化に使っていない新規クエリなので、汎化性能を正直に確認できます。

改善前後の例:

```yaml
# Before
description: Process CSV files.

# After
description: >
  Analyze CSV and tabular data files — compute summary statistics,
  add derived columns, generate charts, and clean messy data. Use this
  skill when the user has a CSV, TSV, or Excel file and wants to
  explore, transform, or visualize the data, even if they don't
  explicitly mention "CSV" or "analysis."
```

改善後説明文は、スキルの処理内容（要約統計、派生列、チャート、クリーニング）が具体化される一方、適用条件（CSV/TSV/Excel、明示キーワードなし）を広くカバーしています。
