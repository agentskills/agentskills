## 実務知識から始める

スキル作成でよくある失敗は、ドメイン固有の文脈を与えずに LLM に作成させることです。一般学習知識だけに依存すると、「エラーを適切に処理」「認証のベストプラクティスに従う」といった曖昧な手順になり、価値の核となる API パターン、エッジケース、プロジェクト規約が抜け落ちます。

有効なスキルは実務知識に根ざしています。鍵は、作成プロセスにドメイン固有の文脈を投入することです。

### 実作業から抽出する

エージェントとの対話で実際のタスクを完了しながら、文脈・修正・好みを与えます。その後、再利用できるパターンをスキル化します。次の点に注目します。

- **うまくいった手順** — 成功につながった行動の順序
- **加えた修正** — エージェント方針を補正した箇所（例: 「YではなくXライブラリを使う」「Zのエッジケースを確認する」）
- **入出力フォーマット** — 入力データと出力データの形
- **与えた文脈** — エージェントが元々知らないプロジェクト固有情報・規約・制約

### 既存成果物から合成する

既存知識が蓄積されているなら、LLM に入力してスキルを合成できます。たとえば、汎用的な「データエンジニアリングのベストプラクティス」記事より、チームの実インシデント報告や runbook から合成したデータパイプラインスキルの方が高性能です。理由は、_自分たちの_ スキーマ、障害モード、復旧手順を含められるためです。重要なのは汎用資料ではなく、プロジェクト固有資料です。

良いソース例:

- 社内ドキュメント、runbook、スタイルガイド
- API 仕様、スキーマ、設定ファイル
- コードレビューコメントと issue tracker（繰り返し指摘と期待値が分かる）
- バージョン管理履歴、とくに修正パッチ（実際に変わった箇所からパターンが見える）
- 実障害ケースと解決手順

## 実行結果で磨く

スキル初稿は通常そのままでは不十分です。実タスクで実行し、失敗だけでなく全結果を作成プロセスに戻して改善します。誤発火の原因、取りこぼし、削れる説明を確認します。

実行→改訂を1回行うだけでも品質は目に見えて向上し、複雑な領域では複数回の反復が有効です。

<Tip>
  最終出力だけでなく実行トレースを確認してください。非生産的な手順に時間を使う場合、原因は「指示が曖昧」「現在タスクに不要な指示を実行」「既定方針なしで選択肢が多すぎる」のいずれかであることが多いです。
</Tip>

テストケース、アサーション、採点を含む構造的な改善方法は [スキル出力品質の評価](/skill-creation/evaluating-skills) を参照してください。

## コンテキストを節約する

スキルが有効化されると、`SKILL.md` 全文が会話履歴・システム文脈・他スキルと同じコンテキストに入ります。スキル内の各トークンは、他情報と注意資源を奪い合います。

### 知らないことだけ足し、既知は省く

スキルがないとエージェントが*知らない*ことに集中してください。プロジェクト規約、ドメイン手順、非自明なエッジケース、使うべきツール/API が対象です。PDF の定義、HTTP の仕組み、DB マイグレーションの一般論は通常不要です。

````markdown
<!-- Too verbose — the agent already knows what PDFs are -->

## Extract PDF text

PDF (Portable Document Format) files are a common file format that contains
text, images, and other content. To extract text from a PDF, you'll need to
use a library. pdfplumber is recommended because it handles most cases well.

<!-- Better — jumps straight to what the agent wouldn't know on its own -->

## Extract PDF text

Use pdfplumber for text extraction. For scanned documents, fall back to
pdf2image with pytesseract.

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
````

各記述について「この指示がないと誤るか？」を自問してください。答えが no なら削除候補です。迷うならテストします。スキルなしでも十分に処理できるなら、そのスキルは価値を追加していない可能性があります。体系的検証は [スキル出力品質の評価](/skill-creation/evaluating-skills) を参照してください。

### 一貫した単位で設計する

スキル範囲の設計は関数設計に似ています。他スキルと組み合わせやすい一貫した作業単位をカプセル化するのが理想です。狭すぎると1タスクで複数スキル読み込みが必要になり、オーバーヘッドや指示衝突が起きます。広すぎると正確に発火しづらくなります。DB 問い合わせ＋結果整形は一単位になり得ますが、DB 運用管理まで含めると広すぎる可能性があります。

### 詳細度は中程度を狙う

過度に網羅的なスキルは逆効果になり得ます。現在タスクに不要な指示に引っ張られ、関連情報の抽出が難しくなるためです。動く例付きの簡潔な手順は、網羅的説明より高性能になりやすいです。すべてのエッジケースを書きたくなったら、多くはエージェントの判断に任せるべきか再検討してください。

### 大きなスキルは段階的開示で分割する

[仕様](/specification#progressive-disclosure) では、`SKILL.md` は500行・5000トークン未満を推奨しています。毎回必要な中核手順だけを残し、追加情報は `references/` などへ分離します。

重要なのは「いつ読み込むか」を明示することです。"詳細は references を見て" より、"API が非200を返したら `references/api-errors.md` を読む" の方が有効です。これにより、段階的開示の設計どおり必要時読み込みができます。

## 指示強度の調整

スキル内の全項目に同じ厳密さは不要です。タスクの壊れやすさに応じて指示の具体度を調整します。

### 壊れやすさに合わせて具体度を決める

**自由度を与える** のは、複数アプローチが有効で結果許容幅がある場合です。柔軟な指示では、厳命より「なぜ」を示す方が有効です。目的を理解したエージェントは文脈依存でより良い判断ができます。コードレビューなら、厳密手順より観点提示が有効です。

```markdown
## Code review process

1. Check all database queries for SQL injection (use parameterized queries)
2. Verify authentication checks on every endpoint
3. Look for race conditions in concurrent code paths
4. Confirm error messages don't leak internal details
```

**厳密に指示する** のは、操作が壊れやすい、再現性が重要、手順順序が必須な場合です。

````markdown
## Database migration

Run exactly this sequence:

```bash
python scripts/migrate.py --verify --backup
```

Do not modify the command or add additional flags.
````

多くのスキルは両者の混在です。部分ごとに個別調整してください。

### メニュー列挙ではなく既定値を示す

複数手段が使える場合、同列に並べるより既定手段を明示し、代替は短く補足します。

````markdown
<!-- Too many options -->

You can use pypdf, pdfplumber, PyMuPDF, or pdf2image...

<!-- Clear default with escape hatch -->

Use pdfplumber for text extraction:

```python
import pdfplumber
```

For scanned PDFs requiring OCR, use pdf2image with pytesseract instead.
````

### 宣言より手順を重視する

スキルは特定ケースの答えではなく、問題群への*アプローチ方法*を教えるべきです。

```markdown
<!-- Specific answer — only useful for this exact task -->

Join the `orders` table to `customers` on `customer_id`, filter where
`region = 'EMEA'`, and sum the `amount` column.

<!-- Reusable method — works for any analytical query -->

1. Read the schema from `references/schema.yaml` to find relevant tables
2. Join tables using the `_id` foreign key convention
3. Apply any filters from the user's request as WHERE clauses
4. Aggregate numeric columns as needed and format as a markdown table
```

これは具体情報を排除する意味ではありません。出力テンプレート（[出力テンプレート](#templates-for-output-format)）、"PII を出力しない" のような制約、ツール固有指示は有効です。要点は、個別情報が具体でも*アプローチ*は汎化可能であることです。

## 効果的な指示パターン

以下はスキル構成の再利用可能テクニックです。すべてを使う必要はなく、タスクに合うものを選びます。

### Gotchas セクション

多くのスキルで最も価値が高いのは gotchas（環境固有で見落としやすい事実）です。これは一般論ではなく、指示しないと起こる誤りへの具体的補正です。

```markdown
## Gotchas

- The `users` table uses soft deletes. Queries must include
  `WHERE deleted_at IS NULL` or results will include deactivated accounts.
- The user ID is `user_id` in the database, `uid` in the auth service,
  and `accountId` in the billing API. All three refer to the same value.
- The `/health` endpoint returns 200 as long as the web server is running,
  even if the database connection is down. Use `/ready` to check full
  service health.
```

gotchas は遭遇前に読める `SKILL.md` に置くのが基本です。別ファイル化する場合は読み込み条件を明示してください。非自明な問題では、トリガーをエージェントが自力認識できないことがあります。

<Tip>
  エージェントの誤りを修正したら、その修正内容を gotchas
  に追加してください。これは反復改善の最短経路の1つです（[実行結果で磨く](#refine-with-real-execution)）。
</Tip>

### 出力テンプレート

出力形式を固定したい場合はテンプレートを提示してください。文章説明より再現性が高く、エージェントは具体構造へのパターン一致が得意です。短いテンプレートは `SKILL.md` に直接書き、長いものや条件付きで必要なものは `assets/` に置いて `SKILL.md` から参照します。

````markdown
## Report structure

Use this template, adapting sections as needed for the specific analysis:

```markdown
# [Analysis Title]

## Executive summary

[One-paragraph overview of key findings]

## Key findings

- Finding 1 with supporting data
- Finding 2 with supporting data

## Recommendations

1. Specific actionable recommendation
2. Specific actionable recommendation
```
````

### 複数ステップ作業のチェックリスト

明示的チェックリストは進捗追跡と手順漏れ防止に有効です。特に依存関係や検証ゲートがある場合に効果があります。

```markdown
## Form processing workflow

Progress:

- [ ] Step 1: Analyze the form (run `scripts/analyze_form.py`)
- [ ] Step 2: Create field mapping (edit `fields.json`)
- [ ] Step 3: Validate mapping (run `scripts/validate_fields.py`)
- [ ] Step 4: Fill the form (run `scripts/fill_form.py`)
- [ ] Step 5: Verify output (run `scripts/verify_output.py`)
```

### 検証ループ

次工程へ進む前に自己検証するよう指示します。基本パターンは「実行→バリデータ実行→修正→再検証」を合格まで繰り返すことです。

```markdown
## Editing workflow

1. Make your edits
2. Run validation: `python scripts/validate.py output/`
3. If validation fails:
   - Review the error message
   - Fix the issues
   - Run validation again
4. Only proceed when validation passes
```

参照ドキュメントをバリデータ代わりにすることも可能です。確定前に参照と照合するよう指示します。

### Plan-validate-execute

バッチ処理や破壊的操作では、中間計画を構造化形式で作成し、正解ソースと照合してから実行させます。

```markdown
## PDF form filling

1. Extract form fields: `python scripts/analyze_form.py input.pdf` → `form_fields.json`
   (lists every field name, type, and whether it's required)
2. Create `field_values.json` mapping each field name to its intended value
3. Validate: `python scripts/validate_fields.py form_fields.json field_values.json`
   (checks that every field name exists in the form, types are compatible, and
   required fields aren't missing)
4. If validation fails, revise `field_values.json` and re-validate
5. Fill the form: `python scripts/fill_form.py input.pdf field_values.json output.pdf`
```

重要なのはステップ3です。計画 (`field_values.json`) を正解ソース (`form_fields.json`) と照合する検証スクリプトが必要です。"Field 'signature_date' not found ..." のような具体エラーは自己修正に十分な情報を与えます。

### 再利用スクリプトの同梱

[スキルを反復改善](/skill-creation/evaluating-skills) する際は、テストケース間で実行トレースを比較してください。毎回同じ処理（グラフ作成、特定形式解析、出力検証）を作り直しているなら、1回テスト済みスクリプトを作って `scripts/` に同梱する合図です。

スクリプト設計と同梱の詳細は [スキルでスクリプトを使う](/skill-creation/using-scripts) を参照してください。
